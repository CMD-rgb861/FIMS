FIMS Role and Permission Refactor - COMPLETION REPORT

Date Completed: May 14, 2026
Status: ALL PRIORITIES COMPLETE ✓

====================================================================
PRIORITY 1 - CRITICAL ✓ COMPLETE
====================================================================

Issue: Backend and frontend did not use one central source of truth for roles and permissions.

Solution Implemented:
1. Centralized role/permission resolution in backend via middleware
   - HandleInertiaRequests.php computes all user flags once
   - Shared via Inertia: id, id_no, role, isAdmin, isDean, isUnitHead, canEvaluateFaculty

2. Created shared role utility helpers (resources/js/utils/role.js)
   - normalizeRole(), isAdminRole(), isUnitHeadRole(), isFacultyRole(), getRoleLabel()
   - All pages now import and use these shared functions

3. Eliminated duplicate permission checks
   - Sidebar, DashboardPage, EvaluationPage, ReportsPage now all use shared utilities
   - Single backend source of truth respected across all components

Result: Frontend and backend perfectly aligned on role/permission state.

====================================================================
PRIORITY 2 - HIGH ✓ COMPLETE
====================================================================

Issue: Frontend trusted raw user fields; redundant canAccessEvaluation prop duplication.

Solution Implemented:
1. Removed redundant canAccessEvaluation prop from sharedUserPayload
   - Simplified to only include canonical user fields
   - Frontend consumes canEvaluateFaculty from user object instead

2. Removed redundant prop from all controller renders
   - DashboardController, EvaluationController, GradesController
   - SubjectsController, ReportsController, AppProfileController
   - All controllers no longer pass separate permission props

3. Maintained backend access gates
   - abort_if(! $canAccessEvaluationForUser(), 403) still in place
   - Backend security boundary remains strong

Result: Single source of truth, no prop sync issues, cleaner backend.

====================================================================
PRIORITY 3 - MEDIUM ✓ COMPLETE
====================================================================

Issue: Controller props were built manually and repeated across files.

Solution Implemented:
1. Created commonInertiaProps() helper in FacultyData trait
   - Builds all common props once: appName, route URLs, csrfToken, user
   - Accepts optional page-specific props to merge

2. Updated all controller methods to use helper
   - DashboardController, EvaluationController, GradesController
   - SubjectsController, ReportsController (index & faculty)
   - AppProfileController (edit & accountSettingsEdit)
   - Eliminated ~80-100 lines of duplicated code

Result: Single source of truth for common props, reduced copy-paste risk.

Implemented a full second-pass optimization of ReportsController.php focused on large dataset performance in index, faculty, and getLocalFacultyUsers.

What was optimized
index() now avoids large full-table get() loads
Replaced full submission load with SQL aggregates for:
total submissions
distinct evaluated instructors
Recent reports now use paginated query (paginate(10)) instead of loading all rows then slicing.
Grade lookup for recent reports is batched and scoped only to rows on the current recent page.
Restored rating_percentage in recent report rows.
Removed accidental duplicate final_grade assignment.
Faculty card SEF ratings are no longer queried per-faculty
Removed per-faculty SupervisorEvaluationSubmission::...->get() calls.
Added batched SEF average computation for all faculty user IDs in one aggregated pass.
Uses SQL JSON aggregation (JSON_TABLE) when supported, with safe cursor fallback if not supported by DB engine.
faculty() now uses paginated external subject query
Subjects are queried directly from lnu_poes.enrollment_courses with pagination (subjects_page, configurable per_page).
Supports school_year_id filtering at query level.
Latest submission and latest grade are fetched with grouped SQL (MAX(id) ... GROUP BY course_code) and then loaded in one batched query each.
Removed heavy all-submission/all-grade per-course in-memory derivation.
getLocalFacultyUsers() now returns lightweight data
No longer loads all subject rows for every faculty.
Uses one grouped query to get only subjects_count per id_no.
Keeps response shape compatible (subjects still present as empty array, plus subjects_count).
Added reusable helpers for rating computation
ratingsToPercent(...)
getUserSefAverageRating(...)
getFacultySefAverages(...)
These centralize computation and include SQL-first + fallback behavior.
